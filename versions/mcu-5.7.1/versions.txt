-->
Enlarging the pool for XML Message Handlers from 1400 to 2000 since it was getting fullwhen connecting 3 Notification Listeners. 
When enlarged, it reached 1495 concurrent allocations.
The general CRc<> mechanism is not robust - it leaves too much work (Reallocate() / Release())in the hands of the user of the reference count, rather than handling it automatically. The above problemis an example for the problems that the current CRc<> implementation could cause - the CRc<SomeInfo>object is Allocated in the EC and then passed to the XML, relying that it will be Released manually after the XML Handler Sends the appropriate message. 
When there was a problem to allocate the Message Handler,the CRc<SomeInfo> was not released. 

-->
Bug fix 56418 - Info objects leak after EventHandler pool ran out.
